Plasma Cylinder

Plasma Cylinder is a small, opinionated WSGI application framework utilizing werkzeug that focuses on keeping web development simple and predictable. Instead of introducing a large stack of abstractions, it builds on Werkzeug’s solid HTTP and WSGI foundations and adds a clear project structure using file-based routing. The goal is to provide more organization than typical microframeworks without the weight of a full-stack framework, making it easy to understand how requests flow through the application while keeping setup and boilerplate to a minimum.

Features

Most Python web frameworks make a tradeoff:

  • micro frameworks stay out of your way, but leave a lot of structure up to each team

  • full-stack frameworks give you structure, but often come with conventions, layers, and abstractions you may not want

This framework is meant to sit in the middle.

It uses werkzeug as the foundation because Werkzeug already solves the hard HTTP problems well: request/response handling, routing primitives, WSGI support, and the general mechanics of web apps. Instead of rebuilding that core, this framework builds on something proven and keeps the stack easy to reason about.

The main opinionated choice is file-based routing.

File-based routing makes the app structure visible from the filesystem. You can look at a project tree and immediately understand what URLs exist and where their handlers live. That lowers the mental overhead for small projects, speeds up onboarding, and makes codebases easier to navigate as they grow.

A route system based on files also reduces boilerplate. You do not need to keep a separate routing table in sync with your views, and you do not need to scatter route declarations across the codebase. The directory structure becomes part of the API surface.

This framework should feel lighter than Django, more structured than Flask, less abstract than many modern async-first frameworks, easy to read, easy to debug, and easy to extend

By staying close to Werkzeug, it keeps the internals understandable. By using file-based routing, it gives developers a simple convention that removes repetitive setup without hiding how requests are handled.

Quickstart

Simply install Plasma Cylinder with pip install cylinder and create the required minimum directory structure.

A minimum viable web app looks something like this:

~/cylinder_sites
    |-- cylinder_main.py
    |-- /my_webapps
        |-- webapp1.ex.get.py

Here is cylinder_main.py:

import cylinder  # pip install cylinder
import waitress # pip install waitress

def main():
    app = cylinder.get_app(app_map)
    waitress.serve(app, host="127.0.0.1", port=80)

def app_map(request):
    # here you can examine the incoming request and decide to route to different webapps
    # depending on the hostname for example

    # you return a tuple of three items:
    # 1) site_dir: the root directory of the site that will handle the request
    # 2) site_name: a meaningful name for the website, perhaps the domain name
    # 3) appended_args: a dict of function arguments and their values that can be used later
    return "my_webapps", "webapp1", {}

if __name__ == "__main__":
    main()

Here is webapp1.ex.get.py:

def main(response):
    response.data = "Hello World!"
    return response

Now if you run cylinder_main.py and navigate to http://127.0.0.1 you should see “Hello World!”.

Implementing “pages”

In the above application, going to http://127.0.0.1 you will see Hello World!, but you would also see Hello World! if you went to http://127.0.0.1/foo/bar

Let’s start by implementing a custom /foo/bar page now:

~/cylinder_sites
    |-- cylinder_main.py
    |-- /my_webapps
        |-- webapp1.ex.get.py
        |-- /webapp1                <==== new directory
            |-- /foo                <==== new directory
                |-- bar.ex.get.py   <==== new file

Here is bar.ex.get.py:

def main(response):
    response.data = "Hello Bar!"
    return response

now navigate to http://127.0.0.1/foo/bar and you will see Hello Bar!

HTTP method routing

You probably noticed the triple-dot file extension .ex.get.py in this case, .ex stands for “executable” (alternatives are .lh and .eh discussed later) and .get represents the HTTP method; a GET request. To write logic for handling the same response to an HTTP POST request, you would create the file bar.ex.post.py.

Custom HTTP methods are also supported, so bar.ex.move.py would be called to process an HTTP MOVE request, despite this not being a standard HTTP method.

Alternatively, a catchall .default can be used. bar.ex.default.py would process any request for which there isn’t a more specific choice:

~/cylinder_sites
    |-- cylinder_main.py
    |-- /my_webapps
        |-- webapp1.ex.get.py           <==== would process HTTP GET to anything except /foo/bar
        |-- /webapp1
            |-- /foo
                |-- bar.ex.get.py       <==== would process HTTP GET to /foo/bar
                |-- bar.ex.post.py      <==== would process HTTP POST to /foo/bar
                |-- bar.ex.default.py   <==== would process HTTP PUT, HEAD, DELETE, Etc. to /foo/bar

The request and response objects

Here’s webapp1.ex.get.py again:

def main(response):
    response.data = "Hello World!"
    return response

Notice we implemented a main() function with a single response parameter. This response object is an empty Werkzeug Response object that’s been created for you and passed into your function. The job of your code is to manipulate it as needed and return it. You can add headers, set the status code, set the body, etc.

But of course, you are driving blind without the corresponding Werkzeug Request object. This is available to you as well as the optional request parameter:

def main(request, response):
    response.data = f"Hello {request.user_agent}!"
    return response

Now if you browse to the page, you’ll see Hello Mozilla/5.0 (...

Note that the order of the parameters on main() don’t matter, so main(response, request) would have worked as well

The parameters on main()

As shown, the process of implementing a new URL path is to create a corresponding file at that path and to implement a main() function. The file at that path is a python module, and the way that main() is called is by importing your module and then running module.main(). It’s the job of your main() function to return the response passed in with any modifications required before it’s to be served to the user agent, so the response parameter is required, but there are a few others that the library will pass in as well if you provide them:

def main(request, response, log, abort, g):             <========= notice all the extra parameters
    log.info("Saying hello to %s", request.user_agent)
    response.data = f"Hello {request.user_agent}!"
    return response

log is the internal cylinder logger. It can be used by your handler modules too as shown above.

abort is the werkzeug abort() function which has had all of the HTTP redirect codes (301, 302, 303, 307, 308) pre-implemented for you as custom errors as well as supporting all of the response codes werkzeug supports naively

g is a SimpleNamespace that’s been implemented with the same rationale as the Flask g object. Its meant to be a scratchpad of sorts. You can use it to store data as a request as it moves through the application, and retrieve it later.

e is an additional parameter not shown above which the framework can sometimes pass into main(), but that’s specific to exception handlers (e is the exception object).

Raising abort() exceptions

As discussed in the werkzeug abort() function link above, the abort() function is just throwing an exception. Those exceptions have default ways of being handled by werkzeug. For example:

def main(request, abort):
    abort(400)

If you navigated to this page, you would see a generic Bad Request page generated by werkzeug:

**Bad Request**
The browser (or proxy) sent a request that this server could not understand.

The abort() function takes an optional description parameter, so you can customize that response a bit like this:

def main(request, abort):
    abort(400, 'Something went wrong')

If you navigated to this page, you would see something like:

**Bad Request**
Something went wrong

But what if you wanted to customize more than this? After all, you can’t write a REST api that returns 400 errors with HTML bodies. That’s where exception handler files come in.

Exception Handlers

To handle abort() exceptions, we create exception handler files using the HTTP status code in the name like so:

~/cylinder_sites
    |-- cylinder_main.py
    |-- /my_webapps
        |-- webapp1.ex.get.py
        |-- webapp1.400.py        <===== will handle all 400 abort()'s
        |-- /webapp1
            |-- /foo
                |-- bar.ex.get.py

to “handle” means that once the abort() is called, the request is now passed into this file’s main() function. From there, you can modify the response as desired to customize it beyond the werkzeug defaults.

All of the error handlers’ main() functions can receive an optional e parameter as well. This is the Exception object raised by the abort() call, and might be useful in debugging or auditing. For example, if you want to write the traceback to the logs, respond with json, you could do this inside of webapp1.400.py

import traceback
import json

def main(request, response, e, log):
    tb_str = ''.join(traceback.format_exception(e))
    log.error('Got a 400 error: %s', tb_str)
    response.content_type='application/json; charset=UTF-8'
    response.data=json.dumps({
        'message': 'bad_request',
        'status': 400,
        'error': True
    })
    return response

Exception handlers are helpful for error-related post-processing tasks as well. For example, if you might want to track the number of 404’s from an IP as part of bot detection and blocking. With a custom 404 handler and a database, that becomes trivial.

In the example above, the entire site uses the same 400 handler, but that’s not required; in fact it would make more sense for 400 errors to output HTML error pages for most of the site, but any 400’s under /api/ to output JSON errors like above.

Implementing multiple handlers based on URL path is done like so:

~/cylinder_sites
    |-- cylinder_main.py
    |-- /my_webapps
        |-- webapp1.ex.get.py
        |-- webapp1.400.py        <===== will handle all 400 abort()'s except the ones under /foo/
        |-- /webapp1
            |-- foo.400.py        <===== will handle all 400 abort()'s under /foo/
            |-- /foo
                |-- bar.ex.get.py

Handling Uncaught Exceptions:

The web at large already has a standard error code to uncaught exceptions: 500 - For that reason, you shouldn’t raise a 500 error using abort(). Instead, use the more specific codes like 501 or 503.

When an uncaught exception is raised, this will make it’s way to your 500 error handler as the e parameter and type werkzeug.exceptions.InternalServerError which contains a .original_exception attribute containing the uncaught exception.

You could use this to do interesting things like send alerts on uncaught exceptions. Here’s an example of a 500 handler that will just output the traceback of all uncaught exceptions in the HTTP response:

~/cylinder_sites
    |-- cylinder_main.py
    |-- /my_webapps
        |-- webapp1.ex.get.py
        |-- webapp1.500.py        <===== will handle all uncaught exceptions
        |-- /webapp1
            |-- foo.400.py
            |-- /foo
                |-- bar.ex.get.py

here’s the content of webapp1.500.py:

import traceback

def main(response, e):
    tb_str = ''.join(traceback.format_exception(e.original_exception))
    response.content_type='text/plain; charset=UTF-8'
    response.data=tb_str
    return response
